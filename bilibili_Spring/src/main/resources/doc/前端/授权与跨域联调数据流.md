# 授权与跨域联调数据流（前后端）

本文描述当前项目里，浏览器请求从发出到返回的授权链路，重点说明：

- 前端跨域预检请求在哪里接收
- 正式业务请求如何走 JWT 鉴权
- `401` 和 `403` 分别由谁返回

## 1. 入口与关键代码位置

- 安全总配置：
  - `src/main/java/com/bilibili/config/security/SecurityConfig.java`
- 路由权限矩阵：
  - `src/main/java/com/bilibili/config/security/SecurityRuleMatrix.java`
  - `src/main/java/com/bilibili/config/security/ApiPaths.java`
- JWT 过滤器：
  - `src/main/java/com/bilibili/security/JwtAuthenticationFilter.java`
- 未登录返回（401）：
  - `src/main/java/com/bilibili/security/RestAuthenticationEntryPoint.java`
- 无权限返回（403）：
  - `src/main/java/com/bilibili/security/RestAccessDeniedHandler.java`
- 方法级权限（资源归属判断）：
  - `src/main/java/com/bilibili/authorization/AuthzService.java`
- CORS 允许域名配置来源：
  - `src/main/resources/db.properties` 的 `cors.allowedOrigins`

## 2. 跨域预检请求（OPTIONS）是怎么被接收的

### 2.1 浏览器何时发预检

当前端发起“非简单请求”时，浏览器会先发一个 `OPTIONS` 预检，例如带了：

- `Authorization` 请求头
- `Content-Type: application/json`
- 或方法是 `PUT/DELETE` 等

### 2.2 后端接收点

预检请求不会先到 Controller，而是先进入 Spring Security 过滤链：

1. `SecurityConfig.configure(HttpSecurity)` 开启了 `.cors()`
2. `SecurityConfig.corsConfigurationSource()` 提供了 CORS 规则
3. `SecurityRuleMatrix` 里有 `OPTIONS /**` 的 `PUBLIC` 规则（放行）

因此，预检请求由安全层处理并直接返回 CORS 相关响应头，不需要业务接口处理。

### 2.3 你现在的 CORS 规则

在 `SecurityConfig.corsConfigurationSource()` 中：

- 允许源：来自 `cors.allowedOrigins`
- 允许方法：`OPTIONS, POST, GET, PUT, DELETE`
- 允许请求头：`Content-Type, Authorization`
- `allowCredentials=false`

如果前端域名不在 `cors.allowedOrigins`，浏览器会在前端侧报 CORS 错误，正式请求不会发出。

## 3. 正式请求的数据流（登录前后）

### 3.1 登录接口（公开）

以 `POST /users/login` 为例：

1. 命中公开路径（`ApiPaths.AUTH_FREE`）
2. 进入 `UserController.login(...)`
3. `JwtTokenService.generateToken(uid)` 生成 JWT
4. 返回 `token` 给前端

前端收到后应自行保存（内存/LocalStorage/SessionStorage 均可，由前端策略决定）。

### 3.2 受保护接口（需要登录）

以 `/me/**` 接口为例：

1. 前端请求头带上 `Authorization: Bearer <token>`
2. `JwtAuthenticationFilter` 读取 `Authorization`
3. 若前缀是 `Bearer `，通过 `substring(7)` 取出 token
4. `JwtTokenService.parse(token)` 解析出 `uid`
5. 构造 `Authentication` 放入 `SecurityContextHolder`
6. 后续授权判断（URL 级 + 方法级）都基于这个登录态

若 token 缺失或解析失败，过滤器会清空上下文，等同未登录。

## 4. URL 级权限与方法级权限

### 4.1 URL 级（SecurityRuleMatrix）

- 公开接口：`/users/login`、`/users/register`、部分 `GET /videos` 等
- 登录可访问：`/me/**`、`POST /users/logout`
- 默认兜底：`anyRequest().authenticated()`

### 4.2 方法级（@PreAuthorize + AuthzService）

例如：

- `@PreAuthorize("@authz.canDeleteComment(authentication, #commentId)")`
- `@PreAuthorize("@authz.canAccessUploadTask(authentication, #uploadId)")`

这里会调用 `AuthzService` 进行“资源归属校验”（是不是本人资源）。

## 5. 401 / 403 的区别与返回位置

- `401 login required`：
  - 场景：未登录或 token 无效，但访问了需要登录的接口
  - 处理器：`RestAuthenticationEntryPoint`
- `403 access denied`：
  - 场景：已登录，但不满足权限规则（例如方法级授权返回 `false`）
  - 处理器：`RestAccessDeniedHandler`

两者都返回统一 JSON 结构 `Result.error(...)`。

## 6. 前端联调最小约定

1. 登录成功后，保存后端返回 token。
2. 调用受保护接口时，统一加头：
   - `Authorization: Bearer <token>`
3. 前端页面域名必须加入 `db.properties` 的 `cors.allowedOrigins`。
4. 如果出现 CORS 报错，先看是否是“浏览器拦截”；如果后端返回 `401/403`，说明请求已到后端。

## 7. 一条完整时序（示例）

1. 前端页面 `http://localhost:63342` 调用 `POST /users/login`。
2. 浏览器先发 `OPTIONS /users/login`（预检）。
3. Security CORS 放行预检，返回允许头。
4. 浏览器发正式 `POST /users/login`，后端返回 token。
5. 前端调用 `GET /me/profile`，携带 `Authorization: Bearer <token>`。
6. JWT 过滤器解析并写入上下文，请求进入业务。
7. 若权限通过返回业务数据；不通过返回 `401/403`。
