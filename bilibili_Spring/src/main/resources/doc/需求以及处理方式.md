# 需求以及处理方式（V1 对齐版）

## 项目节奏提醒（必读）

请优先遵循文档：`开发提醒与演进原则.md`。  
当前阶段只打通 `Spring + Spring MVC + MyBatis-Plus + MySQL`，确保最小闭环可运行。

---

## 1. V1 范围（当前必须完成）

- 项目可启动、数据库可连接、基础接口可联调。
- 完成用户、视频、互动、社交四个核心模块的 MySQL 主链路。
- 返回结构统一，分层清晰，代码可平滑升级到后续分支。

---

## 2. 分层与目录约束

- `controller/`：参数接收、请求校验、统一返回，不写核心业务。
- `service/` + `service/impl/`：业务逻辑与事务控制。
- `mapper/`：MyBatis-Plus 数据访问与必要的 XML 联表查询。
- `model/entity`：与表结构一一对应。
- `model/dto`：请求入参。
- `model/vo`：响应出参。

---

## 3. 数据库口径（以 bilibili.sql 为准）

核心表：

- `t_user`, `t_user_info`
- `t_video`, `t_tag`, `t_video_tag`
- `t_comment`, `t_danmaku`
- `t_video_like`, `t_comment_like`, `t_danmaku_like`
- `t_following`

字段口径提醒：

- 用户主账号字段是 `t_user.username`，不是 `phone`。
- 用户详情表当前不含 `gender` 字段。
- 视频表当前不含 `category/year/publish_time` 字段。
- 评论表当前不含 `deleted` 字段（如需逻辑删除，后续分支再扩展）。

---

## 4. 模块实现方式（V1）

### 4.1 用户模块

- 注册：`username + password + nickname`。
- 登录：账号密码校验，先返回基础登录态（V2 再升级 JWT/Security）。
- 个人页：从 `t_user_info` 返回昵称、头像、签名、粉丝/关注数。

### 4.2 视频模块

- 投稿：上传文件并入库 `t_video(video_url, cover_url, title, description, duration)`。
- 列表/搜索：按标题与状态查询，支持分页。
- 详情：聚合视频信息、作者信息、标签信息与互动状态。

### 4.3 互动模块

- 点赞：使用 like 关系表 + 唯一索引，`status` 做点赞/取消。
- 评论：写入 `t_comment`，用 `parent_id` 支持楼中楼。
- 弹幕：写入 `t_danmaku`，按 `show_time` 读取。

### 4.4 社交模块

- 关注/取关：`t_following` 关系切换。
- 关系判定：无关系 / 单向关注 / 互关。

---

## 5. 状态码统一口径（V1）

- `t_user.status`：`0=正常`，`1=注销`
- `t_video.status`：`0=已发布`，`1=已下架`
- `t_video_like/t_comment_like/t_danmaku_like.status`：`0=已点赞`，`1=已取消`
- `t_following.status`：`0=关注中`，`1=已取关`

---

## 6. V2 增强项（单独分支）

- Redis：排行榜、搜索历史、热点计数缓存。
- 安全框架：Spring Security + JWT。
- 并发增强：异步回写、限流、消息队列。
- 框架升级：迁移到 Spring Boot。

原则：V2 增强不破坏 V1 接口契约与数据库核心字段。

---

## 7. 验收清单（每次迭代自查）

- 是否只使用了 V1 约定字段与表名。
- 是否把业务逻辑放在 `service` 而不是 `controller`。
- 是否保证列表接口分页与稳定排序。
- 是否使用统一返回结构与统一错误处理。
- 是否为未来升级留出了清晰扩展点。
