# 测试代码阅读指南

这份指南基于当前项目 `src/test/java` 的真实结构，目标是让你能快速看懂每类测试在验证什么，尤其是安全包测试。

## 1. 先建立整体地图

测试目录主要分为 5 类：

1. `com.bilibili.controller`  
控制层测试。重点看：HTTP 方法、URL、入参绑定、返回结构、状态码。
2. `com.bilibili.service.impl`  
服务层测试。重点看：业务分支、参数校验、Mapper/Redis 调用、边界处理。
3. `com.bilibili.security`  
安全测试。重点看：JWT、过滤器、401/403、CORS、默认拒绝规则。
4. `com.bilibili.integration`  
集成测试。重点看：Spring 容器是否能启动，真实流程是否可串通。
5. `com.bilibili.task`  
任务测试。重点看：定时任务逻辑、顺序、异常 key 处理。

## 3. 每个测试文件都按这 6 步看

1. 看类注解  
- `@RunWith(MockitoJUnitRunner.class)`：纯单元测试  
- `@RunWith(SpringJUnit4ClassRunner.class)` + `@ContextConfiguration`：集成测试  
- `@WebAppConfiguration`：Web 容器相关集成测试

2. 看成员变量  
- `@Mock`：被替身的依赖  
- `@InjectMocks`：真正被测对象

3. 看 `setUp`  
确认是 `MockMvcBuilders.standaloneSetup(...)` 还是 `webAppContextSetup(...)`。

4. 看测试方法名  
`xxx_should_yyy` 一般是“输入 -> 行为 -> 预期”。

5. 看 AAA 结构  
- Arrange：`when(...)`、构造 DTO  
- Act：调用方法或 `mockMvc.perform(...)`  
- Assert：`assert...` / `andExpect(...)` / `verify(...)`

6. 看 `verify(...)`  
用来确认关键依赖是否被正确调用。

## 4. 安全包测试代码怎么读（详细版）

你现在看不懂安全测试，核心原因通常是：  
“不知道它在测哪一层”。  
先记住这个分层：

1. Token 工具层：`JwtTokenServiceTest`  
2. Filter 层：`JwtAuthenticationFilterTest`  
3. 认证失败/鉴权失败返回层：`RestAuthenticationEntryPointTest`、`RestAccessDeniedHandlerTest`  
4. 全链路规则层：`SecurityConfigIntegrationTest`

### 4.1 全链路先有图，再看代码

请求链路（简化）：

`HTTP Request`
-> `Spring Security Filter Chain`
-> `JwtAuthenticationFilter`（尝试从 Authorization 解析用户）
-> `授权规则（AnonymousRuleContributor/UserRuleContributor + anyRequest().denyAll()）`
-> `Controller`
-> `Result JSON`

如果没登录且接口要求登录：走 `RestAuthenticationEntryPoint` -> 401  
如果已登录但权限不足：走 `RestAccessDeniedHandler` -> 403

### 4.2 `JwtTokenServiceTest`（先看这个）

文件：`src/test/java/com/bilibili/security/JwtTokenServiceTest.java`

阅读重点：

1. `generateAndParse_shouldReturnSameUid`  
验证“生成 token -> 解析 token”后，`uid` 一致。
2. `parse_invalidToken_shouldThrow`  
验证非法 token 会抛异常，不会静默通过。

你在这类里要确认：  
Token 工具本身是否可靠。只有工具可靠，后面 filter 测试才有意义。

### 4.3 `JwtAuthenticationFilterTest`（再看 filter）

文件：`src/test/java/com/bilibili/security/JwtAuthenticationFilterTest.java`

阅读重点：

1. `doFilterInternal_validToken_shouldSetAuthentication`  
当 header 带 `Bearer token` 且解析成功时：  
- `SecurityContextHolder.getContext().getAuthentication()` 不为空  
- principal 的 `uid` 正确  
- `filterChain.doFilter(...)` 一定继续执行

2. `doFilterInternal_invalidToken_shouldKeepAnonymous`  
token 解析失败时：  
- `SecurityContext` 保持空（匿名）  
- 依然放行到后续链路（不会在 filter 内直接中断）

这说明 filter 的职责是“设置身份上下文”，不是直接决定返回 401/403。

### 4.4 `RestAuthenticationEntryPointTest`（401 来源）

文件：`src/test/java/com/bilibili/security/RestAuthenticationEntryPointTest.java`

阅读重点：

1. 调用 `commence(...)` 后，`status == 401`
2. 响应体包含 `code=401` 和 `login required`

这类测试告诉你：401 JSON 的格式是谁输出的，避免前后端对不上。

### 4.5 `RestAccessDeniedHandlerTest`（403 来源）

文件：`src/test/java/com/bilibili/security/RestAccessDeniedHandlerTest.java`

阅读重点：

1. 调用 `handle(...)` 后，`status == 403`
2. 响应体包含 `code=403` 和 `access denied`

这类测试和上一个对称：  
401 是“没登录”；403 是“已登录但无权访问”。

### 4.6 `SecurityConfigIntegrationTest`（最关键）

文件：`src/test/java/com/bilibili/security/SecurityConfigIntegrationTest.java`

这个类在测“组合效果”，不是单个类。

先看它怎么搭环境：

1. `@RunWith(SpringJUnit4ClassRunner.class)` + `@WebAppConfiguration`
2. `@ContextConfiguration(classes = TestConfig.class)`
3. `TestConfig` 里 `@Import(SecurityConfig.class)`，并提供 `UserService/FollowingService/CommentService` 的 mock bean
4. `setUp()` 用的是：
`MockMvcBuilders.webAppContextSetup(webApplicationContext).addFilters(springSecurityFilterChain)`

这几行意味着：  
它测试的是“真实安全过滤器链 + Web 上下文”，不是 standalone 假链路。

再看测试方法的类型：

1. 未带 token 访问受保护接口 -> `401`
- `meProfileWithoutToken_shouldReturn401`
- `meFollowingWithoutToken_shouldReturn401`
- `videoViewWithoutToken_shouldReturn401`
- `createCommentWithoutToken_shouldReturn401`
- `likeCommentWithoutToken_shouldReturn401`

2. 公共接口匿名可访问 -> `200`
- `publicProfileWithoutToken_shouldBeAccessible`
- `listCommentsWithoutToken_shouldBeAccessible`

3. 带 token 时至少不应是 401
- `meFollowingWithToken_shouldNotReturn401`
- `videoViewWithToken_shouldNotReturn401`

这里用“not 401”而不是“必须 200”的原因：  
安全通过后，还可能被控制器/参数校验返回 400/404 等业务码。

4. 默认拒绝规则
- `unknownPath_shouldReturn401ByDefaultRule`

这个与你常见的“未知路径 404”不同。  
当前安全策略是 `anyRequest().denyAll()`，在安全层就被拦下，所以看到 401/403。

5. CORS 预检
- `corsPreflightForLogin_shouldReturn200`

验证了 OPTIONS 预检放行和 `Access-Control-Allow-Origin` 响应头。

## 5. 你最容易混淆的点（专门给你）

### 5.1 为什么有的控制层测试不走真实鉴权？

很多 controller 测试用的是 `standaloneSetup`，并通过  
`TestAuthenticatedUserArgumentResolver` 从 `X-Test-Uid` 注入 `@AuthenticationPrincipal`。  
它是“控制层单测”，不测安全过滤器链。

安全过滤器链是否正确，由 `SecurityConfigIntegrationTest` 专门测。

### 5.2 为什么看到 401，不是 404？

因为请求先过 Security。  
在当前配置里，未命中白名单且未认证，会先被 Security 拒绝，返回 401。

### 5.3 401 和 403 怎么快速判断？

1. 401：通常是未认证（没 token 或 token 无效）
2. 403：通常是已认证但无权限

---

## 6. 安全测试的“最小检查清单”

每读完一个安全测试类，回答这 6 个问题：

1. 这个类测的是哪一层：工具、filter、handler，还是全链路？
2. 是否构建了真实 `springSecurityFilterChain`？
3. 未登录访问受保护接口是否得到 401？
4. 公共接口是否匿名可访问？
5. CORS 预检 OPTIONS 是否放行？
6. 401/403 的 JSON 结构是否稳定？

如果这 6 个都能回答清楚，你就已经读懂这个项目的安全测试了。
