# 开发提醒与演进原则（长期有效）

本文档用于约束项目开发节奏，避免“现在能跑但以后难升级”的问题。  
后续开发、代码评审、AI 协作默认以本文件为准。

---

## 1. 当前阶段目标（第一优先级）

当前只做基础链路打通：

- Spring（非 Spring Boot）
- Spring MVC
- MyBatis-Plus
- MySQL

本阶段目标是形成最小闭环：配置可启动、数据库可连接、基础 CRUD 可跑通、接口可联调。

---

## 2. 暂缓引入内容（后续分支再做）

以下能力先不作为当前主线阻塞项：

- Redis
- Spring Security / JWT
- 限流、异步线程池、MQ
- Spring Boot 改造
- 复杂缓存、复杂监控、复杂中间件

原则：当前阶段优先“可理解、可运行、可迭代”，不追求一次到位。

---

## 3. 为未来升级预留的编码原则

### 3.1 分层必须清晰

- `controller` 只做参数接收、返回组装，不写核心业务。
- `service` 承载业务逻辑。
- `mapper` 只负责数据访问。

### 3.2 面向接口编程

- 优先定义 `service interface` + `service impl`。
- 避免在 Controller 中直接依赖具体实现细节。

### 3.3 模型职责分离

- `DO` 对应数据库表。
- `DTO` 负责请求入参。
- `VO` 负责响应展示。
- 不混用，避免后续改造时牵一发动全身。

### 3.4 配置外部化

- 数据库、文件路径、开关项全部走 `properties`。
- 禁止把环境信息硬编码在业务代码中。

### 3.5 数据库设计留扩展位

- 核心表保留通用状态字段（如 `status`）和时间字段（`create_time` / `update_time`）。
- 关系型操作（点赞、关注）优先唯一索引防重复，便于后续高并发优化。

### 3.6 先实现稳定接口，再做性能增强

- 先保证接口语义稳定、返回结构稳定。
- Redis 与缓存优化放到后续分支做“增强”，不破坏现有接口契约。

---

## 4. 建议的分支演进路线

- `main`：保持可运行、可演示
- `feature/base-mvc-mp-mysql`：当前基础能力迭代
- `feature/redis`：缓存与排行榜、搜索历史
- `feature/security`：登录态与鉴权
- `feature/spring-boot-migration`：向 Spring Boot 迁移

每个分支都要求：可启动、可验证、可回滚。

---

## 5. AI 协作约定（对 Codex 生效）

后续给建议或写代码时，默认遵循：

- 优先服务当前阶段目标，不强推超前技术栈。
- 给出“最小可行实现 + 可演进结构”。
- 如需引入进阶能力，必须明确标注“可选项/下一阶段项”。
- 任何实现都要考虑未来迁移到 Spring Boot、Security、Redis 时的改造成本。

以上约定用于保证：你能边学边做，同时项目不会因为早期写法而在后期重构困难。
