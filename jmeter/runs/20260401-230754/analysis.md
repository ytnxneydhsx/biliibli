# WebSocket 压测与服务端埋点联合分析报告

## 1. 报告范围

本报告基于同一轮 WebSocket 压测的两类信息进行联合分析：

- JMeter 客户端压测结果
- 服务端埋点与连接统计结果

说明：

- JMeter 压测结果来自本次完整结束的运行目录 `D:\biliibli\jmeter\runs\20260401-230754`
- 服务端指标来自同轮压测后提供的埋点快照
- `im.ws.sessions.active` 与 `im.ws.users.online` 属于查询瞬时值，不代表压测期间峰值

## 2. 本轮压测配置

JMeter 脚本：

- 文件：`D:\biliibli\jmeter\bilibili_webscoket_hand.jmx`
- 线程数：`500`
- Ramp-Up：`60s`
- 线程组外层循环：`1`
- 单连接内部心跳循环：`10`
- 心跳间隔：`15000ms`
- WebSocket 读超时：`6000ms`
- 目标地址：`ws://150.158.146.80:8080/ws/im?token=...`

从测试行为看，本轮压测等价于：

- 500 个线程在 60 秒内逐步建立连接
- 每个连接最多尝试发送 10 次心跳
- 最后尝试主动关闭连接

## 5. 联合对齐分析

### 5.1 为什么客户端和服务端数字看起来“不一致”

这两边记录的并不是同一个观察点：

- JMeter 记录的是客户端采样器结果
- 服务端埋点记录的是真正到达服务端后的事件

因此：

- 客户端报错并不一定代表服务端处理代码出错
- 只有真正到达服务端的握手和心跳，才会出现在服务端指标中

### 5.2 握手阶段：两边结论基本一致

JMeter 结果：

- `500` 次建连
- `492` 次成功
- `8` 次失败

服务端结果：

- `494` 次握手尝试
- `494` 次成功
- `494` 次连接建立

联合判断：

- 握手阶段总体是正常的
- 客户端侧有少量连接超时或未成功到达服务端
- 这类差异通常发生在网络、Nginx、连接超时、客户端采样器侧，而不是应用握手逻辑本身

### 5.3 心跳阶段：问题不在“后端 heartbeat 代码错误”

JMeter 结果：

- 心跳采样总数 `5000`
- 仅成功 `135`
- 失败 `4865`

服务端结果：

- `heartbeat.received = 143`
- `heartbeat.ack.sent = 143`
- `heartbeat.ack.failed = 0`

联合判断：

- 服务器对已经收到的心跳全部正常回 ACK
- 所以后端心跳处理逻辑本身没有暴露出明显错误
- 真正的问题是：大量连接没有稳定把心跳送到后端，或者连接在发送心跳前已经失效

这一点和 JMeter 的失败类型完全吻合：

- `Connection reset by peer`
- `Read timed out`
- `there is no connection to re-use`

### 5.4 关闭阶段：关闭失败是结果，不是根因

JMeter 结果：

- `WebSocket Close` 失败率 `100%`

服务端结果：

- `connection.closed = 493`
- 关闭原因大多是异常关闭与不可靠会话关闭

联合判断：

- 关闭采样器失败并不代表服务端没有关闭连接
- 更可能是执行到 `Close` 时，连接已经先一步断开或被服务端清理
- 因此 `Close` 的失败更像“前面连接已经坏了”的连带结果，而不是本轮最核心的问题

### 5.5 关闭原因的含义

从服务端关闭原因看：

- `code_1006 = 334`
  表示大量连接属于异常关闭，通常说明连接并不是按预期正常收尾
- `code_4500 = 158`
  结合代码实现，极大概率对应 `SESSION_NOT_RELIABLE`，即会话因心跳不可靠被清理
- `code_1000 = 1`
  正常关闭仅 `1` 次

联合判断：

- 会话期内存在大量异常断开
- 另有相当一部分连接因心跳未按预期持续到达而被服务端超时清理

## 6. 最可能的问题位置

从本轮联合分析看，问题优先级大致如下：

### 6.1 优先怀疑：客户端压测模型或连接保持方式

证据：

- 握手成功很多，但真正到达后端的心跳很少
- JMeter 侧大量出现“连接被对端重置”与“没有可复用连接”

可能原因：

- JMeter WebSocket 插件在高并发长连接场景下稳定性不足
- 单连接复用方式与插件行为存在边界问题
- 大量线程共用同一个 token，使连接模型不够贴近真实用户行为

### 6.2 次优先怀疑：网络链路或接入层

可能原因：

- 客户端到服务端之间存在公网抖动
- 入口层代理或网络设备对 WebSocket 长连接有额外影响
- 在高并发情况下，连接被中间层提前断开

### 6.3 不优先怀疑：后端 heartbeat 处理代码

原因：

- `heartbeat.received` 与 `ack.sent` 完全一致
- `ack.failed = 0`
- 服务端没有表现出“收到心跳后处理失败”的证据

## 7. 对“当前压测方向是否正确”的判断

当前压测方向是正确的，因为它已经覆盖了两类关键能力：

- WebSocket 握手建连能力
- 建连后的心跳稳定性

这两类指标都非常适合用来做：

- 当前 Spring WebSocket 版本的基线
- 后续 Netty 实现的对照压测

但当前压测模型还有两个明显限制：

- 压测机与服务端看起来不属于严格受控的内网同环境
- 500 连接共用同一个 token，不够贴近真实在线用户模型

## 8. 结论

本轮联合结论可以概括为：

1. 握手阶段基本正常，问题不集中在建连逻辑。
2. 后端 heartbeat 处理逻辑基本正常，收到心跳就能正常回 ACK。
3. 真正的问题发生在会话期，很多连接没有稳定进入持续心跳阶段。
4. 大量失败更像是连接保持问题、客户端插件行为问题、网络链路问题或接入层问题，而不是后端 heartbeat 分支代码报错。

## 9. 下一轮压测建议

建议下一轮按下面顺序做：

1. 将压测环境尽量放到与服务端同地域、同 VPC 或内网环境中，减少公网噪音。
2. 使用多账号、多 token 模式压测，不要 500 连接共用一个 token。
3. 保持当前“线程组只跑 1 轮”的模型，避免无限循环污染结果。
4. 先做分级并发：`20 -> 50 -> 100 -> 200 -> 500`。
5. 同时保留客户端报告与服务端埋点，重点比对：
   - 握手成功数
   - 心跳收到数
   - ACK 成功数
   - 关闭原因分布
6. 如果目标是对比 Spring 与 Netty，务必保持相同场景、相同压测机、相同指标口径。

## 10. 建议用于 Spring 与 Netty 对比的关键指标

建议固定观察以下指标：

- 握手成功率
- 握手 `p95` 与 `p99`
- 心跳成功率
- 心跳 ACK 发送成功率
- 异常关闭比例
- 心跳超时清理比例
- 峰值在线连接数
- JVM CPU / 内存 / GC
- Nginx 或入口层连接数与错误数

---

报告结论一句话版：

本轮压测显示，当前服务端“握手能力正常、心跳处理正常”，但“长连接稳定维持能力”在现有压测链路下表现较差；问题更像发生在会话维持阶段，而不是发生在 heartbeat 业务处理本身。
